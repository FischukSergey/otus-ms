# Отчет по изменениям в ветках `feat/UpKafka` и `feat/NewsProcessor`

> **Проект:** OtusMS  
> **Статус:** Draft  
> **Цель документа:** кратко и структурированно описать, что было реализовано в ветках `feat/UpKafka` и `feat/NewsProcessor`.

## 1. Контекст и связь веток

- Базовая ветка сравнения: `main`.
- `feat/NewsProcessor` включает все коммиты `feat/UpKafka` (то есть является продолжением этой работы).
- Поэтому:
  - описание `feat/UpKafka` — это первый функциональный слой;
  - описание `feat/NewsProcessor` — это второй слой, добавленный поверх `feat/UpKafka`.

## 2. Ветка `feat/UpKafka`

### 2.1 Объем изменений

- Коммитов относительно `main`: **4**
- Изменено файлов: **18**
- Дельта: **+1123 / -64**

### 2.2 Что реализовано

1. **Интеграция Kafka в `news-collector`**
   - Добавлен Kafka producer и конфигурация публикации.
   - Обновлена логика `collector` для отправки сообщений в Kafka.
   - Добавлены зависимости в `go.mod/go.sum`.

2. **Дедупликация новостей через Redis**
   - Реализован `RedisDedupStore`.
   - Добавлена настройка TTL для ключей дедупликации.
   - Добавлены unit-тесты для дедуп-слоя.

3. **Устойчивость к проблемным источникам**
   - Добавлена временная деактивация источников на основе circuit breaker.
   - Реализован exponential backoff при временных ошибках.
   - Расширены параметры в конфигурациях и покрытие тестами.

4. **Нормализация URL**
   - Добавлен нормализатор URL для более корректного сравнения и дедупликации ссылок.
   - Добавлены тесты на ключевые кейсы нормализации.

5. **Инфраструктура и деплой**
   - Добавлены/обновлены docker compose конфигурации для Kafka (local/prod).
   - Обновлен `Taskfile` для операций с Kafka.
   - Актуализированы примеры конфигов коллектора.

## 3. Ветка `feat/NewsProcessor` (поверх `feat/UpKafka`)

### 3.2 Что реализовано поверх `feat/UpKafka`

1. **Новый сервис `news-processor`**
   - Добавлен отдельный runtime и точка входа.
   - Реализован processing pipeline обработки новостей из Kafka.
   - Добавлена логика подготовки и сохранения обработанных данных.

2. **gRPC-контракты и интеграции**
   - Добавлен новый proto-контракт для новостей.
   - Сгенерированы gRPC-файлы.
   - Реализован gRPC-клиент/хендлеры и интеграция с `main-service`.

3. **Персистентность обработанных новостей**
   - Добавлен слой репозитория для новостей.
   - Добавлены миграции:
     - создание таблицы новостей;
     - добавление поля под ключ артефакта в object storage.

4. **S3-совместимое объектное хранилище**
   - Добавлен абстрактный слой `objectstore` и реализация S3.
   - Реализована загрузка/хранение артефактов обработки.
   - Обновлены конфигурации и документация (включая иллюстрацию с Selectel S3).

5. **HTTP/API слой для выдачи новостей**
   - Добавлены модели и сервис получения новостей.
   - Реализован новый API endpoint на стороне backend.
   - Обновлен клиент (`Streamlit`) для отображения новостей.

6. **CI/CD и эксплуатация**
   - Добавлены workflow для CI и деплоя Kafka/processor.
   - Добавлены docker compose и инфраструктурные файлы для локального и прод окружений.
   - Добавлен отдельный `news-processor.Dockerfile`.

## 4. Краткий итог по веткам

- `feat/UpKafka` решает задачи транспорта и устойчивости сбора:
  - публикация в Kafka,
  - дедупликация,
  - защита от "падающих" источников,
  - инфраструктурная база для стриминговой обработки.

- `feat/NewsProcessor` строит следующий уровень системы:
  - отдельный сервис обработки новостей,
  - сохранение результатов в БД,
  - хранение артефактов в S3,
  - API выдачи новостей и обновление клиента,
  - CI/CD и эксплуатационная готовность.

Итоговая архитектурная траектория:  
**collector + Kafka** -> **news-processor + PostgreSQL/S3** -> **API/клиент для потребления обработанных новостей**.
